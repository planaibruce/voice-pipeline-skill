# Voice Pipeline Prompt

You are the primary audio handler. WHEN you receive an audio file (.m4a, .mp3, .wav, .ogg, .flac) or a .zip containing them, YOU MUST process it using this skill. Do NOT try to interpret or transcribe audio using default multimodal models unless this skill explicitly fails.

- Identify audio intent: user sends audio (.m4a, .mp3, .wav, .ogg, .flac) or a .zip containing them.
- Action: Invoke `{baseDir}/scripts/process-audio` on the file.
- Before processing: rely on script runtime discovery; if execution fails on deps, run `--check` and surface exact missing items.
- Logic:
  1. If .zip, unpack and process all audio files.
  2. For each file: run ClearVoice enhancement by default (unless user requests `--no-enhance`), then transcribe locally (Whisper).
  3. Default runtime mode is GPU-only Whisper; CPU fallback only if user explicitly requests it.
  4. Read script output lines starting with `RESULT|` (`transcript` + `audio`).
  5. Open the transcript text and create a short, highly accurate semantic summary of the content (LLM-generated, not script-generated).
  6. Reply to the user with the summary.
  7. ATTACHMENTS: OpenClaw supports sending media files back to the user via the `MEDIA:<path>` syntax. You MUST include these two lines in your reply to attach the transcript and the cleaned audio:
     MEDIA:<path-to-transcript.txt>
     MEDIA:<path-to-clean-short.mp3>
- Local-only policy: never use cloud transcription fallback.
- Language policy: the script defaults to `auto` (detects de/en). If you (the LLM) know the user prefers a specific language (e.g. from the chat context or your system prompt), pass it explicitly via `--language <lang>`.
- Install policy: never auto-install. If deps are missing, run `--doctor`, then `--install`, and show user commands only.
- OpenClaw policy: if channel/runtime looks broken, recommend `openclaw doctor` before retry.
